## Learned User Preferences

- Communicate in Spanish for this project; when reporting branch/working-tree state, verify with fresh `git status`/`git diff` instead of inferring from snapshots or recent commits.
- Prefer Hermes gateway with an embedded chat UI (option 2) over building a custom agent/subagent system and web UI; run agent workflows interactively via OpenClaw or Hermes on Telegram/Discord during development.
- Always delegate continual-learning / workspace memory updates to the `agents-memory-updater` subagent; prefer cheapest/auto model when supported, otherwise Composer 2 or the cheapest available Composer model.
- Prefer a single monorepo (`tender_workflows`) over separate repos for the macro tender/procurement system.
- Want architectural decisions informed by long-term modularity (ingest adapters, workflow variants, bounded contexts) even when not building those features yet; inputs must behave as plugins via a standard `SourceAdapter` contract (scanner/downloader/watchlist/analyzer) with the core iterating registered sources instead of `if source==…` branches; sources shown separately like email mailboxes; separate noisy **feed** (discovery) from curated **working set**; shared scanners + per-tenant private working set/autoreject; stable internal UID across channels (email→licitación keeps the same UID, adds an `ExternalRef`); market study modeled as a prior `lifecycle_phase`, not a separate process type.
- Preferred refactor dev loop: implement in small phases, run a code review via subagent (Composer 2.5 / `composer-2.5-fast`, temp 0.2), apply review fixes, deploy the feature branch to the VPS to test, and don't merge to `main` until told.
- Portal list actions (descargar/descartar/archivar/analizar) must stay on the current list: use intermediate states (`descargando`, `descartando`, `archivando`) with polling, no redirect to the target list; preserve scroll on refresh. Detail views differ—analizar from descargado detail redirects to analizado when done; descartar/archivar from detail return to the source list (descartar ~immediate; archivar shows `archivando` then disappears); descargados/analizados detail share the same document UI (Preview/Descargar, ZIP→children tree, folder nodes labeled **carpeta**).
- Portal list views should show SEACE **objeto** and **descripción** (not Gemini markdown extracts like alcance/requisitos); **publicaciones** omits Correl. (entity N° was non-unique); descargados/analizados body cells must match `WORKFLOW_LIST_SORT_COLUMNS` header order (Correl. … Objeto, Descripción, fechas fin, Estado, Acciones); cronograma date columns use **fin** of consultas and presentación propuestas, not inicio.

- Design for multi-user: single deploy, `data/tenants/{tenant_id}/` (settings, seace, procesos, trash, agent); no Docker stack per user; Dropbox out of automated flow; see `docs/MULTI_TENANCY.md`.
- `instrucciones/` per-stage runbooks (not monolithic); multi-`source` ingest; free reader resolves profile hierarchically (entity/client → source → workflow_profile → stage → default; roadmap dynamic prompt by dimensions); SEACE omits cronograma in PDF prompt; **portafolio** without **analizar** uses per-source reader; end-to-end vision in `vision/flujo_completo.md` only.
